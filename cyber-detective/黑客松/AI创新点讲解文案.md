> **答辩用途** | 更新时间：2026-06-20

> **团队**：A（程序主程）/ B（AI架构师）/ C（美术叙事导演）

## 一、项目概述

**赛博侦探**是一款以 2087 年新香港为背景的赛博朋克侦探叙事游戏，玩家扮演侦探，通过调查场景、收集证据、审讯嫌疑人来侦破案件。项目包含 4 个案件（含 1 个隐藏案件），10 名嫌疑人，12 件证据，完整的跨案件剧情串联。

**核心命题**：如何让 AI 不只是"生成对话"，而是成为**推理引擎**——驱动谎言戳穿、证据组合、压力传导、结局判定等游戏核心机制？

我们的回答是：**双轨架构 + 结构化 Prompt + 动态案件上下文**。

## 二、六大 AI 创新点

### 🔥 创新点 1：双轨引擎架构 —— 在线 AI + 离线预设，零断线体验

**问题**：纯 AI 驱动的游戏一旦 API 不可用，整个审讯/推理系统瘫痪。

**创新**：我们设计了 **"AI 优先 → 预设兜底"** 的双轨回退架构：

```
调用链：interrogation → interrogateAI() → callAPI(混元) → 成功？→ AI 结果
                                                    → 失败？→ fallbackResponse() → 预设数据
```

- **在线模式**：调用混元大模型 API（`hunyuan-lite`），支持指数退避重试（2次，1s/2s间隔）、12s 超时、JSON 格式强制输出
- **离线模式**：63 条预设审讯对话（7嫌疑人 × 3情绪态 × 3-4条）+ 6 组预设证据组合推理 + 3 条预设场景调查

**关键设计决策**：
- 预设数据**不是"随机兜底"**，而是按情绪态精准匹配——`calm` 档回答理性回避，`nervous` 档露出矛盾，`broken` 档坦白真相
- 证据组合兜底采用**双向 ID 匹配**（`relevant_${ev1}_${ev2}` / `relevant_${ev2}_${ev1}`），而非固定顺序依赖
- 离线模式下 **游戏完整可玩**，不会出现"没有可问的问题"或"证据组合无结果"的阻断

**答辩要点**：双轨架构确保了产品级的鲁棒性——即使黑客松现场网络不稳定，演示也不会中断。

### 🔥 创新点 2：谎言戳穿机制 —— 结构化数据驱动的 AI 推理闭环

**问题**：传统侦探游戏的"审讯"是固定对话树，玩家无法用证据戳穿嫌疑人的谎言。

**创新**：我们构建了 **"谎言 → 反证 → 戳穿 → 得分 → 结局"** 的完整闭环：

```
数据层：
  characters.json → lies[].counterEvidence → evidence.json → evidence_002 / evidence_003
                                     ↓
交互层：
  审讯提问 → requiresEvidence 证据解锁过滤 → 出示证据 → AI/Preset 判定 hasContradiction
                                     ↓
状态层：
  hasContradiction → addScore(10~20) → addStress(+10~25) → 压力≥67 → broken → 好结局阈值
```

**具体实现**：

1. **`characters.json` 的 `lies` 数组**：每个嫌疑人定义 2-4 条谎言，每条谎言包含 `topic`（话题）、`lieText`（谎言文本）、`truthText`（真相文本）、`counterEvidence`（能戳穿这条谎言的证据 ID）
2. **`dialogues.json` 的 `requiresEvidence`**：审讯问题通过 `requiresEvidence` 字段关联证据 ID，只有玩家已获取该证据时才能解锁对应问题
3. **AI 引擎的戳穿判定**：Prompt 模板将嫌疑人谎言列表注入 AI 上下文，AI 根据"当前证据 vs 谎言列表"动态判断 `hasContradiction`
4. **离线兜底的戳穿逻辑**：`sceneManager.js` 中 `_generateQuestionResponse()` 检查 `lie.counterEvidence === question.requiresEvidence && evidenceObtained.includes(counterEvidence)`，精准匹配戳穿条件

**答辩要点**：这不是"AI 生成了一段对话"——这是**数据驱动的推理引擎**。谎言、证据、得分、结局四个维度通过结构化数据形成闭环，AI 只是推理闭环中的一个节点。

---

### 🔥 创新点 3：情绪-压力联动模型 —— 三级神经响应，让审讯"有节奏感"

**问题**：传统审讯游戏的"压力条"只是线性数值，没有叙事质感的分层。

**创新**：我们设计了 **"calm → nervous → broken"三级情绪-压力联动模型**：

| 压力范围 | 情绪态 | 行为特征 | AI Prompt 指令 |
|---------|--------|---------|---------------|
| 0-33 | calm | 平稳回避、理性防御 | "说话平稳，像运转正常的终端机。防御是一道防火墙" |
| 34-66 | nervous | 语速加快、矛盾出现、义体出卖 | "系统出现延迟。新谎言补旧谎言。义眼闪烁、散热风扇加速" |
| 67-100 | broken | 崩溃坦白、真相碎片 | "系统崩溃在即。声音颤抖。精英面具剥落。说出真相但不一定全部" |

**Prompt 中的三级指令**（`interrogation.txt`）：

```
▌情绪-压力联动 · 三级神经响应模型

当 currentEmotion = "calm" → 你还在掌控。说话平稳、理性...
当 currentEmotion = "nervous" → 你的系统开始出现"延迟"...
当 currentEmotion = "broken" → 系统崩溃在即。声音颤抖...
```

**答辩要点**：
- 压力值不只是数值——它是**叙事节拍器**。每个压力段位对应不同的 AI 输出风格，审讯有明显的"开局试探 → 中局对抗 → 终局崩塌"节奏
- 情绪态同时驱动**预设数据选择**（7嫌疑人 × 3态 = 63条预设）和**AI Prompt 指令**，双轨在情绪维度上完全对齐
- 游戏机制层也响应情绪态：`nervous` 出现 `hasContradiction` 加分，`broken` 满足好结局阈值（score + stress≥67 双条件触发）

### 🔥 创新点 4：四层穿透式证据组合推理 —— 从"匹配"到"涌现"

**问题**：传统侦探游戏的"证据组合"是简单的 ID 匹配——A+B 永远得出固定结论。

**创新**：我们设计了 **"物理因果 → 人物指向 → 时间逻辑 → 涌现发现"四层穿透推理模型**（`evidenceCombine.txt`）：

```
▌第一层：物理因果链 — 电磁灼烧痕迹 + 改装焊接器 = 精确烧毁神经接口的作案手法
▌第二层：人物指向链 — LXB编号 + 涅槃合同签名 = 资金链：林小北执行 + 赵明月出资
▌第三层：时间逻辑链 — 关闭蜂巢72h → 4分钟安保盲区 → 物理注入蓝屏病毒 = 完整时间线
▌第四层：涌现发现 — 单独看是碎片，组合看恢复出完整缓存记录。钱致远精心设计的保险箱陷阱
```

**数据层支撑**：
- `evidence.json` 的 `canCombineWith` 字段定义组合关系图
- `characters.json` 的 `lies[].counterEvidence` 定义证据指向的嫌疑人
- `cases/case_*.json` 的 `truth.keyEvidence` 定义案件核心证据链

**AI 层增强**：
- Prompt 将"案件背景 + 已有证据 + 待组合证据对"全部注入上下文
- AI 按**方法论指令**逐层推理，而非自由发挥——保证推理有逻辑链而非断言

**离线兜底**：
- 8 组精心撰写的手动推理文本（案件1×2组 + 案件2×3组 + 案件3×3组），每条80-150字，覆盖所有可组合证据对的四层分析
- 双向 ID 匹配：`relevant_201_202` / `relevant_202_201` 都能命中

**答辩要点**：证据组合不是"A+B=C"的简单配对——它是**多维度交叉验证**的推理过程，每层都有独立的判定逻辑。AI 的角色是**推理过程的叙述者**，而非结论的生成器。

### 🔥 创新点 5：动态案件上下文切换 —— 一个引擎驱动 4 个案件

**问题**：多案件游戏需要每个案件独立运转，但 AI 引擎的 Prompt 模板如果硬编码案件1数据，切换案件时需要改代码。

**创新**：我们实现了 **"一个引擎 + 动态数据注入"** 的案件切换架构：

```
sceneManager.setCaseData(caseData)  →  切换 _caseData
                                    →  setAIEngineCaseContext(caseData)  →  切换 AI 引擎上下文
                                    →  caseData.suspects 与 characters.json 自动合并
                                    →  压力值按 suspects 动态初始化
```

**Prompt 模板通用化**（从硬编码到变量注入）：

| Prompt | 硬编码（已移除） | 通用化方式 |
|--------|-----------------|-----------|
| `interrogation.txt` | "林小北不可能说话像企业律师"、"赵明月义眼闪烁" | 按身份动态适配的通用描述 + `{{suspectPersonality}}` / `{{suspectLies}}` |
| `evidenceCombine.txt` | "LXB-003=林小北"、"电磁脉冲→神经接口烧毁" | 方法论描述，零案件引用 + `{{ev1Name}}` / `{{ev2Desc}}` |
| `sceneInvestigate.txt` | "霓虹公寓404室/手术灯/消毒水" | 感官矩阵适配 + `{{sceneName}}` / `{{sceneDescription}}` |
| `ending.txt` | "林小北/赵明月/陈老九/涅槃" | `{{suspectsSummary}}` 动态生成 + `{{caseTruth}}` |

**结局阈值动态化**：

```javascript
// sceneManager.js - _calculateEnding()
const getThreshold = (endingId) => {
  const cond = endings[endingId]?.condition || '';  // "score >= 80"
  const match = cond.match(/score\s*>=\s*(\d+)/);
  return match ? parseInt(match[1]) : null;
};
const goodThreshold = getThreshold('ending_good') ?? 80;  // case_004 → 60
```

**答辩要点**：案件切换不需要改一行代码。`startCase(2)` → 数据自动加载 → 上下文自动切换 → Prompt 自动注入案件2数据 → AI 自动适配案件2的嫌疑人、证据、真相。这是**数据驱动架构**的核心体现。

### 🔥 创新点 6：赛博朋克感官矩阵 Prompt —— 世界观一致性不是事后贴标签

**问题**：AI 生成内容经常"世界观漂移"——偶尔赛博朋克，偶尔现代都市，偶尔奇幻。

**创新**：我们设计了**赛博朋克感官矩阵**的 Prompt 框架，从世界观层面约束 AI 输出：

**审讯 Prompt（`interrogation.txt`）**：
- 角色身份约束："底层角色不会突然说话像企业律师"
- 义体细节约束："义眼不由自主地闪烁、植入体的散热风扇突然加速"
- 场景渗透："霓虹广告牌正巧在闪烁，把他半张脸切成了蓝色"

**场景调查 Prompt（`sceneInvestigate.txt`）** —— **3:2:1:1 感官比例法则**：
- 【视觉 · 3份】：霓虹反光、全息投影残影、冷蓝LED
- 【听觉 · 2份】：服务器嗡鸣、磁悬浮共振、电流声
- 【触觉 · 1份】：地面发黏、金属冰凉、键帽凹陷
- 【嗅觉 · 1份】：焦糊电路+消毒水（诊所）、铁锈+潮湿（废弃空间）、臭氧+合成食品（下城区）

**证据组合 Prompt（`evidenceCombine.txt`）**：
- 推理方法论约束："比对数据哈希值、追踪时间戳、物理痕迹和数字残影交叉验证"
- 术语约束："凶器可能是改装电磁脉冲设备，也可能是远程植入的神经病毒"

**结局 Prompt（`ending.txt`）** —— **三条叙事轨道**：
- 轨道A（好结局）："灰色的黎明，雨停了，霓虹灯还在闪但已经没那么刺眼"
- 轨道B（中等结局）："黄昏时的霓虹灯开始亮起，天空还没完全暗下来"
- 轨道C（坏结局）："深夜暴雨，霓虹灯光被雨水打散成一片模糊的光斑"

**答辩要点**：世界观一致性不是事后检查——它在 Prompt 设计阶段就通过**感官比例法则、术语约束、身份-语气映射、场景类型适配**等机制写入框架。AI 的输出被结构性地约束在赛博朋克世界观内。

## 五、技术亮点速览（30秒答辩版）

1. **双轨引擎**：AI 优先 + 离线兜底，零断线体验，63条预设对话+8组预设推理保证完整可玩

3. **三级情绪-压力联动**：calm/nervous/broken 三档叙事风格，AI Prompt + 预设数据双轨对齐

5. **动态案件切换**：`setCaseData()` 一行代码切换案件上下文，Prompt模板零硬编码
6. **感官矩阵约束**：3:2:1:1比例法则+术语约束+身份语气映射，世界观一致性结构性保证

## 六、为什么这些创新重要？

**传统侦探游戏**的审讯是**固定对话树**——问A永远是回答A，出示证据B永远是回答B。玩家的"推理"本质上是**在预设路径上做选择题**。

**赛博侦探**的审讯是**推理引擎**——玩家的每一个操作（收集证据、组合分析、审讯提问、出示证据、施压）都通过数据闭环影响后续判定。AI 的角色不是"替代预设对话"，而是**在结构化数据框架内做动态推理叙述**——确保每次审讯都有不同的节奏和张力，同时不偏离案件事实。

这就是我们的核心创新：**不是让 AI "生成对话"，而是让 AI 成为推理引擎的叙述层**。
