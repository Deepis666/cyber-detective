### 各模块职责边界

| 模块 | 核心职责 | 不应做的事 |
|------|----------|-----------|
| `gameState.js` | 状态 CRUD、持久化、变更通知 | 不应包含 UI 逻辑 |
| `dialogueSystem.js` | 对话渲染、打字机效果、选项 | 不应包含证据/场景逻辑 |
| `evidenceSystem.js` | 证据栏渲染、选择/组合/出示 | 不应直接操作场景 DOM |
| `aiEngine.js` | API 调用、Prompt 加载、回退 | 不应处理 UI 渲染 |
| `audioManager.js` | 音频预加载、播放、渐入渐出 | 不应处理业务逻辑 |
| `sceneManager.js` | 场景切换、调查热点、审讯、结局 | （当前承担过多，见 §八） |

## 四、数据层架构

### 4.1 数据加载流程

```
main.js
  │
  ├─ fetch('/data/case.json')       → _caseData
  ├─ fetch('/data/characters.json')  → _charactersData
  ├─ fetch('/data/evidence.json')    → _evidenceData
  └─ fetch('/data/dialogues.json')   → _dialoguesData
        │
        ├─ 注入到 window._dialoguesData（⚠ 反模式，见 §八）
        ├─ 合并 CD: _caseData.suspects = [_charactersData.characters[id].find()]
        ├─ 传给 initSceneManager(_caseData.scenes, _caseData)
        ├─ 传给 initEvidenceSystem(_evidenceData)
        └─ 传给 initAIEngine(_charactersData, _evidenceData, _caseData)
```

### 4.2 数据文件 Schema

#### case.json
```jsonc
{
  "caseId": "string",           // 案件唯一 ID
  "title": "string",            // 案件名称（显示用）
  "setting": {                  // 案件背景
    "time": "string",
    "location": "string",
    "background": "string"
  },
  "victim": {                   // 受害者信息
    "name": "string",
    "identity": "string",
    "description": "string"
  },
  "suspects": ["string"],       // 嫌疑人 ID 数组（运行时被 main.js 替换为完整对象）
  "scenes": {                   // 场景字典（key=sceneId）
    "sceneId": {
      "id": "string",
      "name": "string",         // 显示名称
      "description": "string",  // 首次进入场景的旁白

      "hotspots": [{            // 调查热点
        "id": "string",
        "label": "string",      // 悬停提示
        "type": "evidence|move|talk",
        "position": { top, left, width, height }, // 百分比
        "evidenceId": "string",       // type=evidence
        "targetScene": "string",      // type=move
        "once": "boolean"             // 一次性热点
      }]
    }
  },
  "truth": {
    "killerId": "string",       // 真凶 ID
    "motive": "string",
    "weapon": "string",
    "keyEvidence": ["string"]
  },
```


## 六、状态管理架构

### 6.1 设计模式

`gameState.js` 采用 **发布-订阅 + 深拷贝防护** 模式：

- **内部状态** (`_state`)：唯一的可变状态源
- **getState()**：返回深拷贝，外部修改不影响内部
- **updateState(patch)**：局部合并更新
  - 数组字段（evidenceObtained, dialogueHistory）：整体替换
  - 对象字段（stressLevel, flags）：浅合并
  - 标量字段：直接覆盖
- **onStateChange(cb)**：注册变更监听器
- **LocalStorage 持久化**：`saveGame()` / `loadGame()`

### 6.2 运行时状态结构

```js
{
  currentPhase: 'intro'|'investigate'|'interrogation'|'ending',
  currentScene: string|null,        // 当前场景 ID
  currentSuspect: string|null,      // 当前审讯嫌疑人 ID
  evidenceObtained: string[],       // 已获得证据 ID 列表
  evidenceExamined: string[],       // 已查看证据 ID 列表
  dialogueHistory: Array<{role, content, timestamp}>, // AI 对话上下文
  stressLevel: {                    // 嫌疑人压力值 0-100
    suspect_001: number,
    suspect_002: number,
    suspect_003: number
  },
  score: number,                    // 推理评分
  endingTriggered: string|null,     // 触发的结局 ID
  flags: {}                         // 通用剧情标记（key-value）
}
```

## 七、AI 引擎架构

### 7.1 调用流程

```
用户动作 → sceneManager/evidenceSystem
  → aiEngine.interrogateAI() / evidenceCombineAI() / investigateAI()
    → loadPrompt(templateName, variables)    // 加载 .txt 模板并变量替换
    → callAPI(prompt, history)              // 调用混元 API
      ├─ 成功 → 校验 JSON → 返回结果
      └─ 失败 → fallbackResponse(type, ctx) // 离线回退（预设回应池）
```

### 7.2 API 配置

```js
// 通过 Vite 环境变量注入
VITE_HUNYUAN_API_URL  = 'https://api.hunyuan.cloud.tencent.com/v1/chat/completions'
VITE_HUNYUAN_API_KEY  = '' // 为空时自动使用离线模式
```

### 7.3 回退机制

每个 AI 接口都有对应的离线回退：
- **审讯** → `PRESET_RESPONSES.interrogation[suspectId][emotion]` 随机选取预设回应
- **证据组合** → 返回 `isRelevant: false` 的通用提示
- **场景调查** → 返回通用场景描述

## 九、未来扩展约束

### 9.1 不变更原则

以下架构设计在扩展时 **不应修改**，以保证兼容性和稳定性：

1. **模块导入为单向依赖**：保持 `gameState ← 其他模块` 的依赖方向，不允许反向依赖
2. **gameState 深拷贝保护**：`getState()` 的返回值必须为独立副本
3. **AI 离线回退必须保留**：任何新功能都必须实现离线模式回退
4. **Screen 切换通过 CSS class `active`**：不引入路由库或 iframe
5. **证据系统用 ID 引用**：不存储证据对象本身，只存 ID 列表
6. **存档格式不变**：LocalStorage 的 key `cyber_detective_save` 的数据结构向后兼容

## 十、构建与部署

| 配置项 | 值 |
|--------|-----|
| 包管理器 | npm |
| 构建工具 | Vite 6.x |
| 入口 HTML | `index.html`（根目录） |
| 静态资源目录 | `assets/`（配置 `publicDir: 'assets'`） |
| 数据文件 | `data/`（通过 fetch 加载，⚠ 构建到 dist 时需确保 data/ 被复制） |

### ⚠ 注意事项
- `vite.config.js` 中 `publicDir: 'assets'` 意味着 **只有 `assets/` 目录被自动复制到 dist**。
- `data/` 和 `prompts/` 目录不在 publicDir 下，在构建时需要 **额外配置**（如 `vite-plugin-static-copy`）或手动复制。
- 建议：多案件重构时，同时修复 build 配置，确保 `data/` 和 `prompts/` 正确进入 dist。
